# Leone1111_online_notes 产品需求文档 PRD

## 1. 文档信息

- 项目名称：Leone1111_online_notes
- 产品定位：博士生个人学术作品展示与知识沉淀网站
- 目标用户：访问者、学术同行、导师/合作伙伴、招聘方、同学朋友，以及网站所有者本人
- 文档版本：v0.1
- 当前阶段：产品构想与需求定义

## 2. 项目背景

作为一名博士生，用户会持续产生大量有价值的内容，包括读书笔记、科研记录、代码实践、项目成果、论文相关材料、学习路线、公开展示作品等。

这些内容如果只散落在本地文件夹、聊天记录、网盘、Git 仓库或笔记软件里，别人很难系统地了解用户的研究方向、能力结构、成长轨迹和作品价值。

Leone1111_online_notes 希望成为一个面向外部访问者的网页端个人学术作品站。它既是个人主页，也是作品展厅，还是长期知识档案馆。访问者可以通过网站了解用户是谁、正在研究什么、做过什么项目、写过什么笔记、有哪些可展示成果，并通过清晰的入口联系用户。

## 3. 产品愿景

让一个博士生的学习、研究、代码和项目成果，不再只是“文件”，而是变成一个可访问、可理解、可展示、可持续积累的个人知识宇宙。

网站应该像一个安静但有力量的学术工作室：

- 首页让人迅速知道“这个人是谁，研究什么，擅长什么”。
- 作品页让人看到真实成果，而不是空泛介绍。
- 笔记页让人感受到持续学习和思考的过程。
- 项目页让人看到从问题、方法到结果的完整链路。
- 访问者不需要登录，也能顺畅浏览公开内容。
- 用户本人可以方便上传、编辑、分类和展示内容。

## 4. 产品目标

### 4.1 核心目标

1. 建立一个公开可访问的个人学术作品网站。
2. 支持上传和展示读书、科研、代码、项目等不同类型笔记。
3. 让访问者能够快速理解用户的研究方向、能力结构和代表成果。
4. 帮助用户长期沉淀个人知识资产。
5. 支持未来扩展为更完整的个人学术主页、作品集和知识库。

### 4.2 非目标

当前阶段不优先做以下事情：

- 不做复杂社交平台。
- 不做多人协作笔记系统。
- 不做课程售卖平台。
- 不做论文管理软件的完整替代品。
- 不做所有访问者都能上传内容的开放社区。

## 5. 用户画像

### 5.1 网站所有者：博士生本人

需求：

- 上传自己的笔记、项目、代码成果和研究材料。
- 给内容分类、打标签、设置公开或私密。
- 选择代表作品放在首页或作品集页面。
- 用比较体面的方式向别人展示自己。
- 后续可以持续更新，而不是每次都重做网站。

典型场景：

- 准备申请访问学者、博士后、实习、教职或科研合作时，把网站链接发给对方。
- 论文投稿、组会汇报、项目展示后，把补充材料整理到网站。
- 平时读论文、写代码、做实验，把阶段性成果上传归档。

### 5.2 学术同行 / 导师 / 合作者

需求：

- 快速了解用户的研究方向和已有成果。
- 查看代表项目、论文笔记、代码仓库、实验记录。
- 判断是否有合作可能。
- 找到联系方式。

典型场景：

- 通过会议、邮件、论文、GitHub 或朋友推荐进入网站。
- 浏览用户的研究专题和项目页面。
- 下载或查看公开材料。

### 5.3 招聘方 / 评审方

需求：

- 快速判断用户的能力。
- 看到真实项目，而不只是简历文字。
- 理解用户会什么、做过什么、思考深度如何。

典型场景：

- 用户在简历中放入网站链接。
- 招聘方访问首页和代表作品页。
- 查看项目说明、技术栈、成果截图、代码链接和论文相关内容。

### 5.4 普通学习者

需求：

- 阅读公开笔记。
- 学习某个方向的资料整理、论文阅读方法、代码实践过程。
- 收藏或分享有帮助的内容。

典型场景：

- 通过搜索引擎进入某篇笔记。
- 沿着标签继续浏览相关内容。

## 6. 核心使用流程

### 6.1 访问者浏览流程

1. 访问网站首页。
2. 快速看到用户身份、研究方向、代表作品和最新更新。
3. 进入作品集、笔记库、项目页或关于页面。
4. 使用分类、标签、搜索筛选内容。
5. 阅读具体内容详情。
6. 如有兴趣，通过联系方式、邮箱、GitHub、Google Scholar 等入口进一步联系。

### 6.2 用户上传内容流程

1. 用户登录后台。
2. 选择内容类型：读书笔记、科研笔记、代码笔记、项目作品、论文阅读、实验记录等。
3. 填写标题、摘要、分类、标签、封面、正文、附件或外部链接。
4. 设置可见性：公开、仅自己可见、草稿。
5. 预览页面效果。
6. 发布内容。
7. 内容出现在对应栏目、标签页、搜索结果和时间线中。

## 7. 信息架构

网站建议包含以下一级页面：

- 首页
- 作品集
- 笔记库
- 研究专题
- 项目展示
- 代码实践
- 关于我
- 联系方式
- 后台管理

### 7.1 首页

首页是访问者的第一印象，需要回答三个问题：

- 你是谁？
- 你研究什么？
- 我为什么要继续看？

建议模块：

- 个人简介区：姓名/昵称、博士生身份、研究方向、当前机构或身份说明。
- 研究关键词：例如 AI、知识管理、科研工具、软件工程、数据分析等。
- 代表作品：精选 3 到 6 个最能体现能力的作品。
- 最新更新：最近发布的笔记、项目、研究记录。
- 学术轨迹：可以用时间线展示学习、研究和项目经历。
- 快速入口：作品集、笔记库、GitHub、邮箱、简历下载等。

### 7.2 作品集

作品集展示最重要、最成熟、最能对外代表用户能力的内容。

每个作品建议包含：

- 作品标题
- 一句话介绍
- 作品类型：科研 / 代码 / 项目 / 论文 / 工具 / 可视化 / 实验
- 背景问题：为什么做这个作品
- 方法思路：怎么解决
- 最终成果：产出了什么
- 个人贡献：用户本人做了哪些部分
- 图片、视频、链接或附件
- 相关笔记
- GitHub / Demo / PDF / 数据集链接

### 7.3 笔记库

笔记库是长期知识沉淀的核心区域。

内容类型：

- 读书笔记
- 论文笔记
- 科研灵感
- 实验记录
- 代码笔记
- 项目复盘
- 学习路线
- 问题清单

筛选方式：

- 按类型筛选
- 按标签筛选
- 按时间筛选
- 按研究方向筛选
- 按公开程度筛选
- 关键词搜索

笔记详情页建议包含：

- 标题
- 摘要
- 正文
- 标签
- 创建时间和更新时间
- 相关作品或项目
- 参考资料
- 附件

### 7.4 研究专题

研究专题用于把零散内容串成系统脉络。

例如：

- 博士研究主线
- 某一类算法/模型学习笔记
- 某个科研问题的长期追踪
- 某篇论文从阅读到实验复现的完整记录
- 某个项目从想法到实现的全过程

专题页建议包含：

- 专题简介
- 核心问题
- 内容目录
- 阶段性进展
- 相关论文
- 相关笔记
- 相关项目
- 下一步计划

### 7.5 项目展示

项目展示强调“做出来的东西”。

项目类型：

- 网页应用
- 数据分析项目
- 科研工具
- 论文复现项目
- 自动化脚本
- 可视化作品
- 课程项目
- 实验平台

项目详情建议包含：

- 项目背景
- 目标用户
- 核心功能
- 技术栈
- 项目截图
- 难点与解决方案
- 项目成果
- 后续计划
- 代码仓库链接

### 7.6 代码实践

代码实践区用于展示技术能力，但不要只堆代码。

每篇内容建议说明：

- 解决了什么问题
- 使用了什么技术
- 为什么这样设计
- 遇到什么坑
- 最后学到了什么
- 是否有可运行 Demo 或仓库链接

### 7.7 关于我

关于页需要真实、克制、有个人气质。

建议内容：

- 简短个人介绍
- 研究方向
- 教育经历
- 技术能力
- 当前关注的问题
- 代表成果
- 公开链接：GitHub、Google Scholar、ORCID、邮箱、个人简历等
- 一段更有人味的介绍：为什么做研究，为什么做这个网站

### 7.8 后台管理

后台只给网站所有者使用。

核心功能：

- 登录 / 退出
- 内容列表
- 新建内容
- 编辑内容
- 删除内容前二次确认
- 草稿保存
- 发布 / 下架
- 分类管理
- 标签管理
- 附件管理
- 首页精选内容管理
- 访问数据查看

删除规则：

- 删除内容必须二次确认。
- 重要内容建议先进入回收站，不直接永久删除。
- 永久删除前展示清楚标题、类型和影响范围。

## 9. 功能需求

### 9.1 公开访问功能

#### 首页浏览

用户可以访问首页，并看到个人介绍、代表作品、最新内容和主要入口。

验收标准：

- 首页能清楚展示网站主人身份。
- 首页能进入作品集、笔记库、项目展示、关于页面。
- 首页能展示最新发布内容。

#### 内容浏览

访问者可以浏览公开内容。

验收标准：

- 只展示公开内容。
- 点击内容卡片可进入详情页。
- 内容详情排版清晰，适合长文阅读。

#### 搜索和筛选

访问者可以通过关键词、分类和标签查找内容。

验收标准：

- 搜索标题和摘要。
- 支持按内容类型筛选。
- 支持按标签筛选。

#### 分享功能

访问者可以复制内容链接。

验收标准：

- 每篇公开内容有稳定链接。
- 链接可以直接打开对应详情页。

### 9.2 内容管理功能

#### 登录后台

网站所有者可以登录后台管理内容。

验收标准：

- 未登录用户不能进入后台。
- 登录后可以看到管理入口。

#### 新建内容

用户可以创建不同类型内容。

验收标准：

- 可以填写标题、摘要、正文、分类、标签。
- 可以保存为草稿。
- 可以预览。
- 可以发布。

#### 编辑内容

用户可以修改已发布或草稿内容。

验收标准：

- 编辑后保存更新时间。
- 编辑前后内容不应丢失。

#### 删除内容

用户可以删除内容，但必须谨慎。

验收标准：

- 删除前必须二次确认。
- 推荐先进入回收站。
- 永久删除要有明确提示。

#### 首页精选管理

用户可以选择哪些内容显示在首页代表作品区。

验收标准：

- 可以设置或取消首页推荐。
- 首页推荐数量建议限制在 3 到 6 个。

### 9.3 文件上传功能

支持上传：

- 图片
- PDF
- Markdown 文件
- 代码片段附件
- 数据文件说明文档
- 项目截图

限制建议：

- 单个文件大小需要限制。
- 文件类型需要校验。
- 上传失败要有明确提示。
- 私密内容不能通过公开链接访问。

## 10. 页面体验要求

### 10.1 视觉风格

整体风格建议：

- 干净
- 学术
- 现代
- 有个人气质
- 不花哨
- 适合长时间阅读

页面不应该像营销落地页那样夸张，而应该像一个认真做研究的人整理出的开放工作台。

### 10.2 首页气质设想

首页可以像一张“研究者名片”：

- 上方是一句清晰的身份说明。
- 中间是研究方向和代表作品。
- 下方露出最新笔记和项目动态。

示例文案方向：

> 我是一名博士生，正在把读过的书、做过的实验、写过的代码和正在形成的研究问题，整理成一个可被访问的知识空间。

### 10.3 内容阅读体验

长文阅读要舒服：

- 正文字号适中。
- 段落间距清楚。
- 代码块、引用、表格、图片要可读。
- 目录可以自动生成。
- 移动端也能阅读。

### 10.4 移动端体验

网站必须适配手机。

移动端重点：

- 首页信息不能堆太满。
- 导航要简单。
- 内容卡片清楚。
- 长文阅读不能左右横向滚动。

## 11. 权限需求

### 11.1 访问者权限

访问者可以：

- 浏览公开页面。
- 搜索公开内容。
- 查看公开附件。
- 复制公开链接。

访问者不可以：

- 进入后台。
- 查看私密内容。
- 修改内容。
- 删除内容。

### 11.2 管理者权限

管理者可以：

- 登录后台。
- 新建、编辑、发布、下架内容。
- 管理标签和分类。
- 设置首页推荐。
- 上传附件。
- 查看访问统计。

## 12. 安全需求

- 后台必须登录。
- 密码不能明文保存。
- 上传文件需要检查类型和大小。
- 私密文件不能被公开访问。
- 删除操作需要确认。
- 后台接口不能被未登录用户调用。
- 表单输入需要防止恶意脚本。

说明：这里的“恶意脚本”可以理解成别人往网站里塞危险内容，导致访问者或后台受到影响。

## 13. SEO 与传播需求

为了让别人能通过搜索引擎找到网站，公开页面需要支持基础 SEO。

需求：

- 每个页面有清楚的标题。
- 每篇内容有摘要。
- 代表作品和研究专题有可读链接。
- 网站有站点描述。
- 支持被搜索引擎索引公开内容。
- 私密内容不被索引。

可以优化的搜索关键词：

- 用户姓名或昵称
- 博士生个人主页
- 研究方向关键词
- 项目名称
- 论文笔记主题
- 代码实践主题

## 14. 数据统计需求

后台可以查看简单访问数据。

第一阶段建议统计：

- 总访问量
- 热门内容
- 最近访问趋势
- 内容阅读次数
- 来源页面

这些数据帮助用户知道：

- 哪些作品更受关注。
- 哪些笔记能被搜索到。
- 访问者更关心什么方向。

## 15. 运营与内容策略

### 15.1 内容分层

建议把内容分成三层：

第一层：代表作品  
用于展示能力，内容少但质量高。

第二层：系统专题  
用于展示研究脉络，让别人看到持续积累。

第三层：日常笔记  
用于记录过程，不要求每篇都完美。

### 15.2 发布节奏

建议：

- 每周整理 1 到 2 篇笔记。
- 每月更新 1 个专题进展。
- 每季度打磨 1 个代表作品。
- 重要论文、项目和实验完成后及时归档。

### 15.3 内容质量标准

每篇公开内容至少回答：

- 这是什么？
- 为什么重要？
- 我做了什么？
- 有什么结论？
- 别人能从中获得什么？

## 16. MVP 范围

MVP 是第一版最小可用产品，不追求一次做完所有功能。

### 16.1 第一版必须有

- 首页
- 关于我
- 作品集列表
- 笔记库列表
- 内容详情页
- 后台登录
- 新建 / 编辑 / 发布内容
- 标签和分类
- 文件上传
- 公开访问

### 16.2 第一版可以暂缓

- 评论系统
- 多用户系统
- 复杂数据分析
- 订阅通知
- 多语言
- 自动生成摘要
- 与 GitHub / Google Scholar 深度同步

## 17. 后续迭代想象

### 17.1 研究时间线

把用户从博士阶段开始的重要事件串起来：

- 读了什么关键论文
- 做了哪些实验
- 产出了哪些项目
- 形成了哪些研究问题
- 哪些想法后来被推翻或发展

这会让网站不只是展示结果，也展示思考的成长。

### 17.2 个人知识地图

把标签、专题、项目和笔记连接成一张知识图谱。

访问者可以看到：

- 某个研究方向关联了哪些论文。
- 某个项目背后有哪些笔记。
- 某个代码实践服务于哪个研究问题。

### 17.3 AI 辅助整理

未来可以加入 AI 功能：

- 自动提取笔记摘要。
- 自动生成标签。
- 根据已有笔记推荐相关内容。
- 将论文阅读记录整理成结构化卡片。
- 帮助生成项目介绍初稿。

### 17.4 公开简历模式

网站可以生成一个更正式的简历页面：

- 教育经历
- 研究方向
- 论文与项目
- 技术能力
- 获奖经历
- 代表作品
- 联系方式

这个页面可以直接放进求职、合作或申请材料中。

### 17.5 展示模式

为组会、面试、路演或答辩准备一个展示模式：

- 全屏展示代表作品。
- 按时间线讲述项目。
- 每个作品配图和一句话结论。
- 适合现场快速讲解。

## 18. 成功指标

第一阶段成功标准：

- 网站可以公开访问。
- 用户能独立发布内容。
- 至少上传 5 篇笔记。
- 至少上传 3 个代表作品。
- 首页能清楚表达用户身份和研究方向。
- 访问者能在 30 秒内理解网站主要价值。

中长期成功标准：

- 网站成为用户简历、申请、合作沟通中的固定入口。
- 公开内容能被搜索引擎收录。
- 作品和笔记持续增长。
- 用户能通过网站更高效地展示自己。

## 19. 风险与应对

### 19.1 内容太多，结构混乱

应对：

- 一开始就设计分类和标签。
- 代表作品和普通笔记分开展示。
- 首页只展示精选内容。

### 19.2 上传后长期不维护

应对：

- 后台发布流程要简单。
- 支持草稿。
- 允许先发布简版内容，后续再补充。

### 19.3 公开内容不够正式

应对：

- 内容支持草稿和私密。
- 只有打磨过的内容设置为代表作品。
- 普通笔记可以保持自然记录风格。

### 19.4 隐私泄露

应对：

- 默认新内容可设置为草稿或私密。
- 附件访问需要权限控制。
- 发布前提供预览。

## 20. 产品一句话总结

Leone1111_online_notes 是一个为博士生打造的个人学术作品站：它帮助用户把读过的书、做过的研究、写过的代码和完成的项目，整理成一个别人能访问、能理解、能信任的网页端知识空间。
